# 苍穹外卖后端系统
## 一、项目简介
面向餐饮外卖业务场景的前后端分离后端系统，围绕 **管理端运营配置** 与 **用户端下单履约** 两条核心业务链路展开，完成了从员工登录、菜品与套餐管理、分类维护、购物车、地址簿、订单提交、支付状态流转、商家接单、订单派送、订单完结到实时消息提醒的完整闭环。
项目采用分层架构与模块化拆分设计，基于 `sky-common`、`sky-pojo`、`sky-server` 三模块组织公共能力、领域对象与业务服务，具备较强的可维护性、可扩展性与工程化落地能力。
## 二、项目定位
该系统定位于 **餐饮外卖业务中台后端**，重点解决以下几类核心问题：
- **管理端运营问题**：员工身份认证、分类管理、菜品管理、套餐管理、图片上传、订单处理。
- **用户端交易问题**：微信小程序用户登录、地址簿维护、购物车管理、菜品与套餐浏览、订单提交、订单查询、再来一单。
- **履约链路问题**：订单支付成功后的状态推进、商家端待接单提醒、订单接单/拒单/取消/派送/完成流转。
- **系统协同问题**：通过 Redis、缓存、AOP、定时任务、WebSocket 等机制增强系统性能、一致性与实时性。
## 三、核心业务能力
### （1）管理端能力
- 员工登录、退出、分页查询、状态启停、信息编辑、详情查看。
- 分类的新增、分页查询、启停、修改、删除以及按类型检索。
- 菜品的新增、分页查询、回显、修改、批量删除、起售/停售、按分类查询。
- 套餐的新增、分页查询、数据回显、修改、批量删除、起售/停售。
- 基于阿里云 OSS 的文件上传能力，用于图片资源统一存储与访问。
- 商家端订单管理能力，包括条件搜索、待处理统计、详情查询、接单、拒单、取消、派送、完成。
### （2）用户端能力
- 基于微信小程序 `code -> openid` 的用户登录链路。
- 菜品与套餐浏览，支持按分类查询与口味/套餐明细展示。
- 地址簿维护，支持默认地址设置与多地址管理。
- 购物车管理，支持菜品/套餐加入、数量累加、列表查询、清空购物车。
- 订单提交、历史订单分页查询、订单详情、取消订单、再来一单。
- 订单支付后的提醒能力，以及用户侧催单能力。
- 商户营业状态查询。

### （3）订单履约链路

系统打通了完整的订单状态流转：

`待付款 -> 待接单 -> 已接单 -> 派送中 -> 已完成`

同时支持异常分支：

- 用户取消订单
- 商家拒单
- 管理端取消订单
- 超时自动取消订单

这一设计使项目不仅覆盖了基础 CRUD，更具备真实业务系统所要求的 **状态机驱动型订单处理能力**。

## 四、支付链路说明

本项目在支付环节采用了 **“模拟支付联调方案”**：

- 后端保留了微信支付相关的配置模型与 SDK 依赖，具备后续接入真实微信支付的扩展基础。
- 当前实现中，为避免开发联调阶段对外部支付网关的强依赖，系统通过 `/user/order/payment` 直接完成订单支付成功态推进。
- 支付接口调用后，系统会将订单状态由 **待付款** 推进为 **待接单**，同时将支付状态更新为 **已支付**，并通过 WebSocket 向管理端广播新订单提醒。

这种实现方式的价值在于：

- **优先验证核心履约链路**，先打通“下单-支付成功-商家接单-配送-完成”的主流程。
- **降低外部依赖复杂度**，避免在开发阶段被第三方支付证书、回调通知、签名校验阻塞主线业务推进。
- **为真实支付升级预留接口**，后续可以无缝替换为正式微信支付网关而不破坏业务边界。

换言之，这一版本的重点不是“支付网关对接本身”，而是 **订单支付完成后的业务编排能力与商家履约闭环能力**。

## 六、关键技术实现
### （1）双端 JWT 鉴权隔离
项目对 **管理端** 与 **用户端** 分别配置独立的 `secretKey`、TTL 与请求头名称，实现了双端鉴权隔离：
- 管理端走 `/admin/**` 拦截链路。
- 用户端走 `/user/**` 拦截链路。
- 通过拦截器完成 JWT 解析与身份校验。
- 解析成功后，将当前登录主体 ID 写入 `BaseContext(ThreadLocal)`，在 Service 层直接获取。
  这种方式避免了在 Controller → Service → Mapper 的调用链中层层传递用户 ID，提升了代码整洁度与业务聚合性。
### （2）AOP 自动公共字段填充
针对新增与修改操作，项目通过自定义注解 + 切面方式自动填充：
- `createTime`
- `updateTime`
- `createUser`
- `updateUser`
  该方案将审计字段维护逻辑从业务代码中抽离，避免了重复编写模板代码，使 Mapper 调用更聚焦于业务本身，体现了较强的横切逻辑治理能力。

### （3）Redis 缓存与序列化治理

项目自定义了 `RedisTemplate`，采用：

- `StringRedisSerializer` 处理 Key/HashKey
- `GenericJackson2JsonRedisSerializer` 处理 Value/HashValue

从而解决默认 JDK 序列化导致的可读性差、跨语言不友好、调试困难等问题。

同时项目在用户端菜品查询处使用 `@Cacheable`，在管理端菜品新增/修改/删除/起售停售处使用 `@CachePut` 和 `@CacheEvict` 完成缓存刷新与失效控制，形成了清晰的 **读缓存、写删缓存** 策略。

### （4）购物车并发控制

购物车新增能力中，项目针对同一用户的并发添加场景，采用了：

- **基于用户 ID 的本地串行化锁**：`synchronized(userId.toString().intern())`
- **事务方法通过代理对象调用**：确保锁内逻辑仍可命中 Spring 事务代理

这说明项目并非停留在简单接口开发层面，而是已经考虑到了同一用户短时间重复点击导致购物车数量竞争的问题，具备一定并发一致性意识。

### （5）订单链路事务封装

订单提交流程通过事务统一控制，核心过程包括：

1. 校验地址簿是否存在。
2. 校验购物车是否为空。
3. 写入订单主表。
4. 批量写入订单明细表。
5. 清空用户购物车。
6. 返回订单提交结果。

该流程体现的是典型的 **主子表事务落库模型**，确保订单主记录、订单明细与购物车清理之间具备原子性。

### （6）模拟支付 + WebSocket 实时通知

在模拟支付实现中，系统不仅修改订单状态与支付状态，还会通过 WebSocket 主动向管理端推送 JSON 消息，实现：

- 新订单支付成功提醒
- 用户催单提醒

这使管理端能够第一时间感知订单变化，强化了系统的实时交互能力，也让该项目从传统的“请求-响应式 CRUD”升级为具备 **事件通知能力** 的业务系统。

### （7）定时任务驱动的订单补偿治理

项目通过定时任务对订单生命周期进行自动治理：

- 每分钟扫描超时未支付订单，自动取消。
- 每日凌晨扫描长时间未人工完结的派送中订单，自动置为已完成。

这一能力本质上属于 **基于时间窗口的业务补偿机制**，是业务系统稳定性设计的重要组成部分。

### （8）批量查询与 N+1 问题规避

在历史订单与管理端订单查询场景下，项目并未逐条订单查询明细，而是：

- 先收集当前页订单 ID
- 再批量查询所有订单明细
- 最后按 `orderId` 分组回填

该做法有效规避了典型的 **N+1 查询问题**，对分页列表接口的数据库压力控制更加友好，也能体现较好的数据访问优化意识。

### （9）业务规则保护

在分类与菜品删除逻辑中，系统引入了明确的业务前置校验，例如：

- 分类被菜品或套餐关联时禁止删除
- 菜品处于起售状态时禁止删除
- 菜品被套餐关联时禁止删除

这类设计体现了对 **领域约束一致性** 的重视，而不仅仅是“数据库有数据就直接删”的粗放式处理。

## 七、工程亮点总结

### （1）具备完整的交易闭环设计能力

项目不是零散接口拼接，而是围绕“用户下单-支付成功-商家处理-派送完成”建立了完整业务闭环，具备清晰的订单生命周期治理能力。

### （2）具备一定的并发与一致性处理意识

购物车并发控制、事务化订单提交流程、缓存失效策略、ThreadLocal 上下文隔离，说明项目已经开始关注高频业务场景下的数据正确性问题。

### （3）具备实时化系统设计能力

通过 WebSocket 实现商家端订单提醒与催单通知，增强了业务响应时效性，体现了从“离线型管理系统”向“实时协同系统”的能力延伸。

### （4）具备良好的工程拆分与横切治理能力

模块化拆分、统一鉴权、AOP 自动填充、统一上下文传递、接口文档生成、配置外置化等能力，说明项目在工程组织层面具备较好的规范化意识。

### （5）具备外部平台集成与扩展基础

系统已接入微信小程序登录、阿里云 OSS，并预留微信支付配置与 SDK 依赖，具备进一步向真实生产链路扩展的基础条件。

## 八、简历项目描述

基于 Spring Boot 2.7 + MyBatis + Redis + WebSocket 实现餐饮外卖业务后端，采用模块化分层架构支撑管理端运营与用户端交易场景。围绕订单主链路实现了事务化下单、模拟支付状态推进、WebSocket 实时催单/新订单提醒、超时关单与自动完结任务；同时通过 JWT 双端鉴权、ThreadLocal 上下文透传、AOP 公共字段填充、缓存失效治理与批量查询规避 N+1 等机制提升系统工程质量与业务可维护性。
